## 1. 方案概述

### 1.1 平台定位

构建一个**统一、通用、可扩展**的企业级大数据平台，支撑以下核心能力：

```
┌──────────────────────────────────────────┐
│          企业大数据平台核心能力             │
│                                          │
│   ✅ 全域数据采集      ✅ 统一数据存储     │
│   ✅ 批流一体计算      ✅ 规范数据仓库     │
│   ✅ 自助数据分析      ✅ 实时数据服务     │
│   ✅ 数据治理管控      ✅ 智能数据应用     │
└──────────────────────────────────────────┘
```

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **统一性** | 一个平台覆盖所有数据场景，避免烟囱式建设 |
| **可扩展** | 架构分层解耦，各层可独立扩展和替换 |
| **批流一体** | 同一套架构同时支持实时和离线，减少维护成本 |
| **自助化** | 业务人员能自己查数据、做报表，减少对开发的依赖 |
| **成本可控** | 按需建设，分期投入，小规模也能跑起来 |
| **安全合规** | 数据分级管控，满足安全审计和合规要求 |

### 1.3 适用场景

本方案适用于但不限于以下场景：

| 场景 | 示例 |
|------|------|
| 经营分析 | 营收报表、用户增长分析、转化漏斗 |
| 实时监控 | 业务大屏、异常告警、实时指标 |
| 用户画像 | 标签体系、人群圈选、精准营销 |
| 日志分析 | 系统监控、错误排查、性能分析 |
| 数据服务 | 数据 API、数据产品、数据共享 |
| 智能应用 | 推荐系统、风控模型、预测分析 |

### 2.2 架构设计说明

| 层级 | 职责 | 核心原则 |
|------|------|----------|
| **采集层** | 对接所有数据源，统一汇入平台 | 全量 + 增量，不丢数据 |
| **传输层** | 数据管道，解耦上下游 | 高吞吐、低延迟、可回溯 |
| **存储层** | 分类存储，各取所长 | 湖仓一体，冷热分离 |
| **计算层** | 批流一体，统一加工 | SQL 优先，降低使用门槛 |
| **服务层** | 统一对外提供数据能力 | 统一入口，权限管控 |
| **应用层** | 业务消费数据 | 自助化，低代码 |
| **基础设施层** | 调度、监控、治理、安全 | 全平台支撑，保障稳定 |

## 3. 技术选型方案

### 3.1 采集层选型

| 场景 | 推荐技术 | 备选 | 选型理由 |
|------|----------|------|----------|
| 数据库实时同步 | **Canal** (MySQL) | Debezium（多数据库） | Canal 对 MySQL 最成熟，Debezium 更通用 |
| 日志文件采集 | **Filebeat** | Flume | 轻量级，和 ELK 生态一致 |
| 批量数据导入 | **DataX** | Sqoop（已淘汰） | 支持数据源最多，阿里开源，社区活跃 |
| API / 文件 / 复杂流 | **NiFi** | 自研 | 可视化编排，非开发人员也能配置 |

### 3.2 传输层选型

| 组件 | 推荐 | 选型理由 |
|------|------|----------|
| 消息总线 | **Apache Kafka** | 吞吐最大、生态最好、大数据标配，所有数据统一走 Kafka |

> **设计要点**：Kafka 作为全局数据总线，所有实时数据先进 Kafka，再分发到下游各存储/计算引擎。实现数据的**一次采集、多次消费**。

### 3.3 存储层选型

| 存储类型 | 推荐技术 | 存什么 | 选型理由 |
|----------|----------|--------|----------|
| 数据湖（主存储） | **HDFS/S3 + Apache Iceberg** | 全量原始数据 + 数仓分层表 | 湖仓一体，支持多引擎，Schema Evolution |
| OLAP 分析库 | **Apache Doris** | 高频查询的汇总数据 | 兼容 MySQL，上手简单，实时写入好 |
| 搜索引擎 | **Elasticsearch** | 日志、全文搜索 | 日志分析标配 |
| KV 缓存 | **Redis** | 实时指标、用户标签、热数据 | 毫秒级响应 |

> **为什么选 Iceberg 而不是 Hudi/Delta Lake？**
> - 引擎兼容性最好：Spark、Flink、Trino、Doris 都支持
> - 完全开放，不绑定特定厂商
> - 社区增长最快，长期看最安全的选择

> **为什么选 Doris 而不是 ClickHouse？**
> - 兼容 MySQL 协议，团队学习成本低
> - 自动管理分布式，运维更简单
> - 支持实时更新和多表 Join
> - ClickHouse 单表性能更强，但运维门槛高

### 3.4 计算层选型

| 计算类型 | 推荐技术 | 选型理由 |
|----------|----------|----------|
| 离线计算 | **Apache Spark** | 批处理生态最丰富，Spark SQL 成熟 |
| 实时计算 | **Apache Flink** | 真正的流处理，毫秒级延迟，Exactly-Once |
| 交互式查询 | **Trino** + **Doris** | Trino 联邦查询跨数据源，Doris 秒级 OLAP |
| SQL 模型管理 | **dbt** | 版本化管理 SQL 转换逻辑，可测试、可复用 |

> **批流一体策略**：
> - 数据湖（Iceberg）同时支持 Spark 批写入和 Flink 流写入
> - 上层统一用 SQL 开发（Spark SQL + Flink SQL）
> - 减少两套代码维护成本

### 3.5 服务层选型

| 组件 | 推荐技术 | 职责 |
|------|----------|------|
| 统一查询 | **Trino** | 跨数据源联邦查询，一个 SQL 查多个库 |
| 数据 API | **自建网关** 或 **DataService** | 把数据封装成 RESTful API 给业务系统调用 |
| 缓存加速 | **Redis** | 高频查询结果缓存，降低后端压力 |

### 3.6 应用层选型

| 场景 | 推荐技术 |
|------|----------|
| 数据报表 / BI | **Apache Superset** |
| 实时监控大屏 | **Grafana** 或 **DataV** |
| 自助查询 | **Trino + Superset** |
| 机器学习 | **Spark MLlib + MLflow** |

### 3.7 基础设施选型

| 能力 | 推荐技术 | 选型理由 |
|------|----------|----------|
| 任务调度 | **Apache Airflow** | 全球最流行，DAG 编排灵活 |
| 集群管理 | **Kubernetes** | 容器化部署，弹性伸缩 |
| 服务协调 | **ZooKeeper** | Kafka/HBase 等依赖 |
| 监控告警 | **Prometheus + Grafana + Alertmanager** | 指标→可视化→告警一条龙 |
| 日志分析 | **ELK Stack**（ES + Logstash + Kibana） | 平台自身日志分析 |
| 元数据管理 | **DataHub** | 现代化数据目录，数据血缘 |
| 数据质量 | **Great Expectations** | SQL 式规则校验 |
| 权限安全 | **Apache Ranger** | 统一权限管控 |

## 4. 数据流转设计

### 4.1 实时数据链路

```
数据源（MySQL/日志/API）
    │
    ▼
采集（Canal / Filebeat）
    │
    ▼
Kafka（统一消息总线）
    │
    ├──▶ Flink 实时计算 ──▶ Doris（实时分析）──▶ Grafana/大屏
    │
    ├──▶ Flink 实时计算 ──▶ Redis（实时指标）──▶ 业务系统
    │
    ├──▶ Flink 实时计算 ──▶ Iceberg（实时入湖）──▶ 数仓
    │
    └──▶ Kafka → ES（日志链路）──▶ Kibana
```

**关键设计点**：

| 要点 | 说明 |
|------|------|
| Kafka 统一入口 | 所有实时数据先进 Kafka，一次采集多次消费 |
| Flink 处理 | 数据清洗、转换、聚合、窗口计算 |
| 多路输出 | 同一份数据可以同时写入 Doris（查询快）、Iceberg（存储全）、Redis（实时取） |
| Exactly-Once | Flink + Kafka 端到端精确一次，不丢不重 |

### 4.2 离线数据链路

```
数据源（MySQL/文件/API）
    │
    ▼
批量采集（DataX / NiFi）
    │
    ▼
数据湖 HDFS/S3 + Iceberg（ODS 原始层）
    │
    ▼
Spark 离线计算（清洗 → 建模 → 聚合）
    │
    ├──▶ Iceberg DWD/DWS 层（明细 + 汇总）
    │
    ├──▶ Doris ADS 层（应用数据，给报表用）
    │
    └──▶ Redis / API（给业务系统用）
    
    ▼
Airflow 调度（每天定时跑，依赖管理）
    │
    ▼
Superset 报表 / Trino 自助查询
```

**关键设计点**：

| 要点 | 说明 |
|------|------|
| T+1 定时调度 | Airflow 每天凌晨跑昨天的数据 |
| 分层加工 | ODS → DWD → DWS → ADS，逐层提炼 |
| 存算分离 | 数据存 Iceberg，计算用 Spark，互不影响 |
| 结果推送 | 汇总结果推到 Doris/Redis，加速查询 |

### 4.3 数据回流链路

> 处理后的数据需要回流到业务系统

```
数仓 ADS 层
    │
    ├──▶ 数据 API ──▶ 业务系统（按需查询）
    │
    ├──▶ Redis 缓存 ──▶ 业务系统（高频访问）
    │
    ├──▶ MySQL 回写 ──▶ 业务后台（运营使用）
    │
    └──▶ Kafka 推送 ──▶ 下游系统（事件驱动）
```

## 5. 数据仓库设计

### 5.1 分层架构

```
┌───────────────────────────────────────────────────────┐
│                                                       │
│   数据源  ──▶  ODS  ──▶  DWD  ──▶  DWS  ──▶  ADS    │
│   (原始)      (贴源)     (明细)     (汇总)     (应用)  │
│                                                       │
│   越往右：数据越干净、越聚合、越接近业务                   │
└───────────────────────────────────────────────────────┘
```

| 层级 | 全称 | 数据特征 | 加工逻辑 | 存储位置 |
|------|------|----------|----------|----------|
| **ODS** | Operational Data Store | 原始数据，不做改动 | 直接同步，保留原貌 | Iceberg |
| **DWD** | Data Warehouse Detail | 清洗后的明细数据 | 去重、去空、标准化、脱敏 | Iceberg |
| **DWS** | Data Warehouse Summary | 按主题汇总的宽表 | 聚合、关联、计算指标 | Iceberg + Doris |
| **ADS** | Application Data Store | 直接给应用用的数据 | 面向场景加工 | Doris + Redis |

### 5.2 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| ODS 表 | `ods_{来源}_{表名}` | `ods_mysql_user_order` |
| DWD 表 | `dwd_{主题}_{描述}` | `dwd_trade_order_detail` |
| DWS 表 | `dws_{主题}_{粒度}_{周期}` | `dws_trade_store_1d` |
| ADS 表 | `ads_{应用}_{描述}` | `ads_report_daily_revenue` |
| 维度表 | `dim_{描述}` | `dim_user_info`, `dim_product` |
| 临时表 | `tmp_{描述}` | `tmp_order_dedup` |

### 5.3 主题域划分（参考）

| 主题域 | 包含内容 | 典型指标 |
|--------|----------|----------|
| **用户域** | 用户注册、登录、画像 | DAU、MAU、留存率 |
| **交易域** | 订单、支付、退款 | GMV、客单价、转化率 |
| **商品域** | 商品信息、库存、分类 | SPU/SKU 数量、动销率 |
| **流量域** | PV、UV、页面路径 | 跳出率、停留时长 |
| **营销域** | 活动、优惠券、推广 | ROI、核销率 |

> 💡 主题域需要根据实际业务来定，以上为电商参考，其他行业类似展开

推荐方案：星型模型（主）+ 宽表（辅）

                │  order_id     │

### 5.5 数据质量规则

| 规则类型 | 示例 | 检查时机 |
|----------|------|----------|
| **非空检查** | user_id 不能为空 | ODS → DWD |
| **唯一性检查** | order_id 不能重复 | DWD |
| **范围检查** | amount > 0 且 < 1000000 | DWD |
| **时效性检查** | T+1 数据不能超时 | ODS 入库时 |
| **一致性检查** | DWS 汇总 = DWD 明细之和 | DWS 产出后 |
| **波动检查** | 今日数据量与昨日偏差 < 30% | 每层产出后 |

## 8. 高可用与容灾

### 8.1 高可用设计

| 组件 | 高可用方案 | RPO / RTO |
|------|-----------|-----------|
| **Kafka** | 3 副本，跨机架部署，ISR 机制 | RPO=0, RTO<30s |
| **HDFS** | 3 副本，NameNode HA（Active-Standby） | RPO=0, RTO<60s |
| **Flink** | Checkpoint + Savepoint，自动恢复 | RPO=0, RTO<120s |
| **Doris** | 3 副本，FE HA（1 Master + 2 Follower） | RPO=0, RTO<30s |
| **Redis** | 主从 + Sentinel 或 Cluster 模式 | RPO≈0, RTO<10s |
| **Airflow** | HA Scheduler + Celery Worker 多节点 | RTO<60s |
| **ES** | 多节点集群，1 副本 | RPO=0, RTO<60s |

> RPO = 数据最多丢多少，RTO = 多久能恢复服务

### 8.2 数据备份策略

| 数据 | 备份方式 | 频率 | 保留时间 |
|------|----------|------|----------|
| 数据湖（Iceberg） | HDFS 快照 + 跨集群复制 | 每天 | 30 天 |
| Doris | 全量备份 + 增量备份 | 每天全量 + 每小时增量 | 7 天 |
| Redis | RDB 快照 + AOF | 持续 | 3 天 |
| Kafka | Topic 多副本 + MirrorMaker | 实时 | 按 Retention 保留 |
| 元数据（Airflow/DataHub） | 数据库备份 | 每天 | 30 天 |

### 8.3 容灾等级

| 等级 | 方案 | 成本 | 适合场景 |
|------|------|------|----------|
| **L1 基础** | 单集群多副本 + 本地备份 | 低 | 中小企业 |
| **L2 同城** | 同城双机房，异步复制 | 中 | 大型企业 |
| **L3 异地** | 异地多活，实时同步 | 高 | 金融/核心业务 |

> 💡 建议：大部分企业 L1 就够了，先做好多副本 + 定期备份，别过度设计

### 8.4 故障应急预案

| 故障场景 | 影响 | 应急措施 | 恢复时间 |
|----------|------|----------|----------|
| Kafka Broker 宕机 | 实时链路短暂中断 | 自动 Leader 切换，消费者重连 | < 30s |
| Flink 任务挂掉 | 实时计算中断 | 从 Checkpoint 自动恢复 | < 2min |
| HDFS NameNode 故障 | 离线存储不可用 | HA 自动切换到 Standby | < 60s |
| Doris FE 故障 | 查询服务中断 | HA 自动切换 Follower | < 30s |
| Airflow 调度故障 | 离线任务不触发 | HA Scheduler 切换 + 手动补数 | < 5min |
| 整集群断电 | 全部服务不可用 | 按优先级逐步恢复 | < 30min |

### D. 项目风险评估

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 团队经验不足 | 进度延迟、架构走偏 | 高 | 引入外部顾问 / 培训 |
| 数据源复杂 | 采集困难、质量差 | 中 | 先聚焦核心数据源，逐步扩展 |
| 业务需求变更 | 返工 | 高 | 分期建设，快速迭代 |
| 性能不达标 | 体验差 | 中 | 提前压测，预留扩容空间 |
| 预算超支 | 项目暂停 | 中 | 从小规模起步，按需扩展 |

---

> 📄 本方案为通用架构，实际落地时需根据具体业务场景、数据规模和团队情况进行调整。
>
> 💡 核心原则：**先跑起来，再优化；先小规模，再扩展；先解决核心问题，再追求完美。**
