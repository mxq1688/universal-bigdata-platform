## 采集（数据怎么来）

| 技术 | 用途 | 状态 |
|------|------|------|
| **Kafka** | 实时消息管道，最核心 | 🟢 主流 |
| **Flume** | 服务器日志采集 | 🟡 成熟偏老 |
| **Canal** | 监听 MySQL 变更实时同步 | 🟢 主流 |
| **Debezium** | CDC 数据变更捕获，支持多种数据库（比 Canal 更通用） | 🟢 主流 |
| **DataX** | 多数据源批量搬运（阿里开源） | 🟢 主流 |
| **Sqoop** | Hadoop 与关系型数据库之间批量传输（经典） | 🔴 过时，被 DataX 替代 |
| **NiFi** | 可视化数据流编排，拖拽式操作 | 🟢 成熟 |

## 消息队列（数据怎么传）

> Kafka 是王者，但不是唯一选择

| 技术 | 特点 | 状态 |
|------|------|------|
| **Kafka** | 吞吐量最大，生态最好，大数据标配 | 🟢 主流 |
| **RocketMQ** | 阿里出品，事务消息强，电商/金融场景多 | 🟢 主流 |
| **Pulsar** | 存算分离架构，多租户，云原生友好 | 🔵 新兴上升 |
| **RabbitMQ** | 轻量级，功能全，中小系统够用 | 🟢 成熟 |

> 💡 选型：大数据管道首选 Kafka，业务消息可选 RocketMQ，云原生方向看 Pulsar

## 存储（数据存哪里）

| 技术 | 适合场景 | 状态 |
|------|----------|------|
| **HDFS** | 海量文件分布式存储 | 🟢 成熟，但被对象存储蚕食 |
| **HBase** | 百亿行按 ID 秒查 | 🟡 成熟偏老，新项目少用 |
| **ClickHouse** | 统计分析极快（列式） | 🟢 主流 |
| **Apache Doris** | 国产 MPP 分析引擎，简单易用，兼容 MySQL 协议 | 🔵 新兴热门 |
| **StarRocks** | 极速 OLAP，实时分析性能强（Doris 分支演进） | 🔵 新兴热门 |
| **Elasticsearch** | 全文搜索 + 日志分析 | 🟢 主流 |
| **Redis** | 内存缓存、实时计数 | 🟢 主流 |
| **TDengine** | IoT 时序数据 | 🟢 细分主流 |

> 💡 OLAP 选型：ClickHouse 生态最成熟，Doris/StarRocks 对 MySQL 用户更友好，按团队情况选

## 数据湖 / 湖仓一体（新一代存储）

> 传统数仓太死板，数据湖来兜底，湖仓一体是趋势

| 技术 | 用途 | 状态 |
|------|------|------|
| **Delta Lake** | Spark 生态数据湖（Databricks 出品） | 🟢 主流 |
| **Apache Iceberg** | 开放表格式，多引擎通用，社区最活跃 | 🔵 主流上升最快 |
| **Apache Hudi** | 增量数据处理 + 数据湖，适合频繁更新场景 | 🟢 主流 |

三者对比：Iceberg 最开放通用，Delta Lake 和 Spark 绑定最深，Hudi 对更新场景最友好

## 数据仓库分层（数据怎么组织）

> 数据不是一锅炖，要分层加工，越往上越干净

```
原始数据 → ODS → DWD → DWS → ADS → 报表/应用
```

| 层级 | 名称 | 作用 |
|------|------|------|
| **ODS** | 原始数据层 | 原封不动存进来 |
| **DWD** | 明细数据层 | 清洗、去重、标准化 |
| **DWS** | 汇总数据层 | 按主题聚合统计 |
| **ADS** | 应用数据层 | 直接给报表/接口用 |

### 数据建模方法

| 模型 | 思路 | 适合场景 |
|------|------|----------|
| **星型模型** | 一个事实表 + 多个维度表，简单直接 | 大多数 BI 报表场景 |
| **雪花模型** | 维度表再拆分，更规范 | 数据规范要求高的场景 |
| **宽表** | 把所有字段拍平到一张大表 | 查询性能优先，空间换时间 |
| **Data Vault** | Hub + Link + Satellite，灵活可追溯 | 企业级数仓，变化频繁的业务 |

> 💡 实际项目中：大部分用星型模型 + 宽表就够了，不要过度设计

## 常见架构

**实时**：`Kafka → Flink → ClickHouse → Grafana`

**离线**：`DataX → HDFS → Spark + Hive → Superset`

**湖仓一体**：`Kafka → Flink → Iceberg → Trino → Superset`

**小团队**：`MySQL → Grafana`（够用就行）

### 经典架构模式

| 架构 | 思路 | 优缺点 |
|------|------|--------|
| **Lambda** | 批处理 + 流处理并行，结果合并 | 最全面，但要维护两套代码 |
| **Kappa** | 全部走流处理，历史数据也重放 | 架构简单，但对流引擎要求高 |
| **湖仓一体** | 数据湖 + 数据仓库合二为一 | 新趋势，灵活且统一 |

## 数据中台 vs 数据仓库 vs 数据湖

> 这三个概念经常搞混，一张表说清楚

| 概念 | 本质 | 特点 | 适合 |
|------|------|------|------|
| **数据仓库** | 结构化数据的集中存储 | 数据先建模再存，Schema-on-Write | 报表、BI 分析 |
| **数据湖** | 原始数据全量存储 | 先存再用，Schema-on-Read | 数据探索、机器学习 |
| **数据中台** | 组织能力 + 技术平台 | 不只是技术，更是数据服务化体系 | 多业务线共享数据 |
| **湖仓一体** | 数据湖 + 数据仓库合一 | 既灵活又规范，当前主流趋势 | 通吃 |

> 💡 数据仓库解决"怎么分析"，数据湖解决"怎么存"，数据中台解决"怎么共享"

## 高频概念速查（面试 / 汇报必备）

| 概念 | 一句话解释 |
|------|-----------|
| **CAP 定理** | 一致性、可用性、分区容错，三选二（分布式系统基本约束） |
| **数据倾斜** | 数据分布不均，少数节点干了大部分活 → 解决：打散 Key、加盐 |
| **Exactly-Once** | 数据不丢不重，精确处理一次（Flink/Kafka 都支持） |
| **数据血缘** | 一条数据从哪来、经过哪些处理、到了哪里 |
| **SLA** | 服务等级协议，如 99.9% 可用 = 一年最多宕 8.76 小时 |
| **ETL vs ELT** | ETL 先转换再加载，ELT 先加载再转换（数据湖时代偏 ELT） |
| **维度建模** | 按业务主题建模（谁买了什么、什么时候、在哪里） |
| **事实表 vs 维度表** | 事实表存指标（金额、数量），维度表存描述（用户、商品、时间） |
| **分区与分桶** | 分区按日期等切目录，分桶按 Hash 切文件，都是为了查得快 |
| **Checkpoint** | Flink/Spark 的容错机制，定期快照，挂了能恢复 |
| **背压（Backpressure）** | 下游处理不过来，告诉上游慢点发 |
| **窗口（Window）** | 流处理中按时间/数量切一段数据来计算（滚动/滑动/会话窗口） |
| **Watermark** | 处理乱序数据的机制，"我愿意等你多久" |
| **幂等性** | 同一操作执行多次结果一样（重试安全） |
| **Schema Evolution** | 表结构变了（加列/改类型），数据湖要能兼容处理 |

> 💡 面试三板斧：数据倾斜怎么解决 + Exactly-Once 怎么保证 + Flink 和 Spark 区别

## 🆚 核心技术对比

### Flink vs Spark

| 维度 | Flink | Spark |
|------|-------|-------|
| 核心模型 | 流处理为主，批是流的特例 | 批处理为主，流是微批 |
| 实时性 | 真正逐条处理，毫秒级延迟 | 微批处理，秒级延迟 |
| 状态管理 | 原生支持，非常强 | 较弱，依赖外部存储 |
| Exactly-Once | 原生支持，端到端 | 需配合外部事务 |
| SQL 支持 | Flink SQL，持续优化中 | Spark SQL，非常成熟 |
| 生态 | 实时场景强，湖仓方向发力 | 批处理/ML/数据湖生态最丰富 |
| **选型建议** | **实时计算首选** | **批处理 / 数据湖 / ML 首选** |

### ClickHouse vs Doris vs StarRocks

| 维度 | ClickHouse | Apache Doris | StarRocks |
|------|-----------|-------------|-----------|
| 架构 | 单机性能极强，集群需手动管 | MPP 架构，自动管理 | MPP 架构，自动管理 |
| 上手难度 | 较高，运维复杂 | 低，兼容 MySQL 协议 | 低，兼容 MySQL 协议 |
| 实时写入 | 一般，适合批量导入 | 好，支持实时更新 | 非常好，实时性能最强 |
| 查询性能 | 单表聚合极快 | 多表 Join 友好 | 多表 Join + 实时都强 |
| 社区 | 国际化，最成熟 | 国内活跃，百度→Apache | 国内热门，Doris 分支 |
| **选型建议** | **极致单表分析性能** | **MySQL 用户、简单易用** | **实时分析、高并发查询** |

### Iceberg vs Hudi vs Delta Lake

| 维度 | Iceberg | Hudi | Delta Lake |
|------|---------|------|-----------|
| 引擎兼容 | 最好，Spark/Flink/Trino 通吃 | 好，偏 Spark | 强绑定 Spark |
| 实时写入 | 支持 | 最好，CDC 场景强 | 支持 |
| 社区趋势 | 最活跃，增长最快 | 稳定 | Databricks 主导 |
| 开放性 | 完全开放 | 开放 | 部分功能需 Databricks |
| **选型建议** | **通用首选，多引擎场景** | **CDC / 频繁更新场景** | **全 Spark 生态** |

### Kafka vs RocketMQ vs Pulsar

| 维度 | Kafka | RocketMQ | Pulsar |
|------|-------|----------|--------|
| 吞吐量 | 最高 | 高 | 高 |
| 延迟 | 毫秒级 | 毫秒级 | 毫秒级 |
| 事务消息 | 支持 | 最强 | 支持 |
| 多租户 | 弱 | 一般 | 原生支持，最强 |
| 运维难度 | 中等 | 较简单 | 较复杂 |
| 生态 | 最丰富 | 国内生态好 | 云原生方向 |
| **选型建议** | **大数据管道首选** | **电商/金融业务消息** | **云原生/多租户场景** |
